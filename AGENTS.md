# AGENTS.md

Windows-only game-automation bot for **Magia Exedra** (game window title `MadokaExedra`). Image-recognition clicking (OpenCV `TM_SQDIFF_NORMED` + PyAutoGUI) with a PySide6 GUI. Code comments and GUI prose are Chinese-first, while technical identifiers and game-mode proper nouns may remain in English. `README.md` is the Chinese source; `README_EN.md` and `README_JP.md` are maintained translations.

## Running / building

- Run: `python main.py` (launches the Qt window; `if __name__ == '__main__'` sets `os.chdir` to the script/exe dir so all relative paths resolve). On startup `mywindow.__init__` and the switcher's `_refresh()` call `language_switcher.ensure_active()`, which auto-creates the `aim/` junction if it is missing/broken.
- Build exe: `pyinstaller -D -i resource/main.ico main.py` (documented inline in `main.py`, near the imports). When frozen, `OPENCV_SKIP_PYTHON_LOADER=1` is set near the top of `main.py` before the delayed `src.workers` import eventually reaches `cv2` through `src.click.click_action` -> `src.click.click_behavior`. `sys._MEIPASS`/`sys.executable` determine the base path; in frozen mode `get_executable_directory()` (`main.py`) / `src/packs/image_scaler.base_dir()` / `src/packs/language_switcher.base_dir()` all return `os.path.dirname(sys.executable)`; in script mode all three resolve to the repo root.
- **DPI-sensitive import order:** `src.workers` is intentionally imported lazily by `main.get_worker_registry()`, not at module import time. The startup order must remain `QApplication(...)` -> `mywindow()` -> `get_worker_registry()`, because importing workers imports PyAutoGUI, whose initialization may set the Windows process DPI mode. Importing workers before `QApplication` causes Qt's `SetProcessDpiAwarenessContext()` call to fail with access denied. Do not move `from src.workers import ...` back to the top level.
- **No `requirements.txt`** - runtime source dependencies are `pyautogui`, `PySide6`, `opencv-python` (`cv2`), `numpy`, `pywinctl`, and `Pillow` (required by PyAutoGUI/PyScreeze screenshots). `opencv-python` normally installs NumPy transitively, but this project imports NumPy directly. Building additionally requires `pyinstaller`. `src/packs/image_scaler.py` shells out to the committed `tools/ImageMagick/magick.exe` or falls back to `magick` on `PATH`.
- **No tests, lint, typecheck, or CI.** Verify by running the bot against the live game.

## Architecture (GUI separated from run logic)

The run/automation logic lives in the `src/workers/` package; `main.py` is **GUI only** - it constructs the worker threads, sets their parameters, and starts/stops them. Each mode's flow is in its own file and can be edited independently.

- `main.py` - GUI entry point only. `mywindow(QWidget)` (Qt GUI, title `圆哆啦挂机器`, icon `./resource/main.ico`, initial `resize(250,740)`) + `LanguageSwitcherWidget(QWidget)` (the language/resolution switcher UI). **It does not hardcode any worker** - after `QApplication` exists, it calls the lazy `get_worker_registry()` and builds one worker instance and parameter page for each `WorkerMeta`. A script `QComboBox` selects the current worker, a `QStackedWidget` shows only that worker's parameters, and one shared start button starts the selected entry. The parameter stack has a fixed vertical size policy; `_script_changed()` resets its fixed height to the selected page's `sizeHint()` so shorter pages do not leave a large blank area. If the registry/selection is invalid, the shared button displays `没有可用的挂机脚本` and is disabled. `_start_worker(entry)` first refuses concurrent automation/scaling, then calls `click_behavior.find_win('MadokaExedra')`; if the window is missing, cannot be focused/read, or has invalid dimensions, it logs the failure and does **not** start the worker. It then calls `click_action.detect_window_resolution()` and logs the result; a mismatch beyond the tolerant margin only emits a warning and does **not** block the start. It then reads each parameter getter, `setattr`s values onto the worker, logs `start_hint`, disables automation/template controls, and calls `worker.start()`. The stop button calls `worker.stop()` on every entry. `closeEvent()` requests all workers to stop, cancels scaling, waits up to 5 seconds for each worker and the scaling thread, and rejects closing if background work has not stopped safely. `检查更新` is currently a reserved log-only entry because no version source is configured; it remains clickable while automation/scaling controls are disabled. **There are no module-level parameter globals anymore**. `LanguageSwitcherWidget`: dropdown labels are Chinese (`英语`/`日语` via `language_switcher.lang_label()`, not a GUI-side table), item data holds the real code (`EN`/`JP`); selecting an item auto-switches via `QComboBox.activated`. The only template-management button is `刷新列表`, which re-scans packs and runs `image_scaler.scale_all()` in a background `threading.Thread`, piping progress to the log via signals. GUI state makes switching/scaling and automation mutually exclusive within one process. `main.py` contains no automation flow code, no `QThread` subclass, and no run-flag or parameter globals.
- `src/workers/` package - the automation run logic, fully decoupled from the GUI:
  - `src/workers/registry.py` - the **registration system**. `ParamSpec` describes one worker parameter (`key`, `label`, `kind`, `default`, `min`, `max`, `choices`, and optional `hint`); `hint` is currently stored but not rendered by `main.py`. `kind` is `'choice'` (radio group), `'lp_recover'` (min/-1/input/+1/max widget, auto +1 to storage value), or `'int'` (plain integer, reserved). `WorkerMeta` describes one mode (`name`, `label`, `worker_class`, `params`, `start_hint`). `@register(...)` appends a `WorkerMeta` to the global `REGISTRY`; `get_registry()` returns a copy. **Adding a new bot mode = write a no-argument-constructible `BaseWorker` subclass file, decorate it with `@register`, and add one import line in `src/workers/__init__.py`** - the script selector and parameter stack then include it automatically; no `main.py` edits.
  - `src/workers/base.py` - `BaseWorker(QThread)` base class + `retry_until()` + the `RETRY_TIMEOUT` (60s) / `BATTLE_TIMEOUT` (1800s) constants. Encapsulates run/stop state: `_active` (bool, set by the worker itself; turned `False` on internal stop such as timeout / LP exhaustion) + `_stop_event` (`threading.Event`, set by `stop()` from the GUI). `_running()` = `_active and not _stop_event.is_set()` (same semantics as the old `guaji==1 and not stop_event`). `start()` is overridden to reset state so one instance can be restarted; `stop()` is safe to call on a not-yet-started/already-finished thread. `_click_until()` retries a template-group click and self-stops on timeout; `_wait(s)` wraps the stop event. Emits `signal(str)` for logging.
  - `src/workers/link_raid.py` - `LinkRaidWorker(BaseWorker)` (was `WorkThread_1`), registered as `'link_raid'` with params `level_choice` (choice lv6–lv12, default 6) and `lp_recover_times` (lp_recover 0–10, default 0). The whole link-raid loop: `into_link_raid` -> `link_raid_to_backup_requests` -> `prepare_battle` -> `check_join_full` -> `clean_full` -> `find_lv` -> `join_battle` -> `check_already_end` -> `battle_and_finish`. First navigation uses hardcoded 2K-baseline coords via `click_position_scaled`.
  - `src/workers/crystalis.py` - `CrystalisWorker(BaseWorker)` (was `WorkThread_2`), registered as `'crystalis'` with param `lp_recover_times` (lp_recover 0–8, default 8). The crystalis loop: `click_play` -> `wait_win` -> `click_retry_or_recover_lp`.
  - `src/workers/__init__.py` - re-exports `BaseWorker`, `LinkRaidWorker`, `CrystalisWorker`, `retry_until`, `RETRY_TIMEOUT`, `BATTLE_TIMEOUT`, plus the registration API (`register`, `ParamSpec`, `WorkerMeta`, `REGISTRY`, `get_registry`). Importing the worker submodules here triggers their `@register` decorators. In the GUI process this package import is deliberately delayed until after `QApplication` construction.
- `src/click/` package - the click layer (workers call `from src.click import click_action`; `click_action` imports `click_behavior` via `from . import`):
  - `src/click/click_action.py` - high-level actions. `click_item_with_result` / `find_item_with_result` iterate numbered template groups; `click_position(x,y)` and `move_a_to_b(...)` perform raw window-relative coordinate clicks/drags with window-bounds validation; `click_position_scaled(x,y)` / `move_a_to_b_scaled(...)` auto-scale **2K-baseline** coordinates to the active resolution. They read `language_switcher.current_selection()` + `image_scaler.scale_factor()` and multiply by 0.5/0.75/1.0/1.5 so the same 2K coordinates target the same UI at 720p/1080p/2K/4K. `detect_window_resolution(tolerance=0.1)` compares the game window's client-area size (from `click_behavior.get_client_size()`) against the active pack's nominal resolution (parsed from `language_switcher.current_selection()`'s `res` string), allowing `max(40, round(expected*tolerance))` px of slack per dimension to absorb title-bar/border/DPI drift; it returns a dict `{detected, expected, expected_res, matched, message}` and never raises.
  - `src/click/click_behavior.py` - low-level. `get_xy` captures only the `MadokaExedra` window in memory, runs OpenCV `cv2.matchTemplate(..., cv2.TM_SQDIFF_NORMED)`, and reports `match_rate = 1 - sqrt(min_val)`; threshold **>0.8** is hardcoded in `routine`/`routine_only_find`. Missing windows, invalid screenshots/templates, and oversized templates return a normal no-match result instead of raising. `find_win(title)` is separate from the `2`/`1` action-result convention: it restores/activates the first matching window and returns `(left, top, width, height)` on success, or `None` if lookup/focus/geometry validation fails. `get_client_size(title='MadokaExedra')` returns the window's client-area `(width, height)` (the actual render area, excluding title bar/borders) via pywinctl's `getClientFrame()`, or `None`; it does not activate/restore the window, so callers should ensure the window is shown (e.g. via `find_win`) first. `click_auto` does move-then-click (a raw click is rejected by the game), and `routine` rechecks the worker stop callback immediately before clicking.
  - `src/click/__init__.py` - re-exports the `click_action` / `click_behavior` submodules.
- `src/packs/` package - template-pack management, both pure-stdlib (no Qt) and runnable standalone (`from src.packs import language_switcher, image_scaler`):
  - `src/packs/language_switcher.py` - manages the `aim/` junction: `list_packs()`, `current_selection()`, `switch(lang,res)`, `ensure_active()`, `pack_usable()`. Also holds the `LANG_LABELS` dict and `lang_label(code)` helper (the Chinese display name for a language code) so the GUI reads labels from here instead of keeping its own copy. Reads/writes `language/active.json` (`{lang, res}`). Runnable standalone for debugging.
  - `src/packs/image_scaler.py` - scales the `*_2560x1440` source pack to other resolutions using `tools/ImageMagick/magick.exe` (or `magick` on PATH). Each derived pack keeps `.source_hashes.json`; a target is skipped only when the source SHA-256, resize recipe, and current target SHA-256 all match the manifest. Missing, source-changed, recipe-changed, or locally modified targets are atomically regenerated; previously managed targets deleted from the source are removed. Incomplete source scans skip cleanup. Source walks never follow junctions/reparse points; derived paths reject traversal, absolute paths, Windows reserved names, and reparse-point parents. Images and manifests use temporary files followed by `os.replace`. The scaler accepts `.png`, `.jpg`, `.jpeg`, `.gif`, and `.webp`, but runtime numbered templates and `pack_usable()` are PNG-based. Never writes into the `*_2560x1440` source pack. Runnable standalone for debugging.
  - `src/packs/file_lock.py` - Windows cross-process template-management lock. `template_write_lock()` creates a repository-specific named mutex derived from the normalized real base path. `language_switcher.switch()` uses it with a 2-second timeout; `image_scaler.scale_all()` uses it with a 30-second timeout and cancellation support. It serializes switch/scale writes across multiple running instances; workers do not hold this lock while reading `aim/`.
  - `src/packs/__init__.py` - re-exports the `language_switcher` / `image_scaler` submodules.

Flow: `main.py` (GUI) starts `src/workers/` (the modes in `REGISTRY`) -> `src/click/click_action` (template iteration + coords) -> `src/click/click_behavior` (match + click + window). Separately, `main.py` 刷新列表 -> `src/packs/image_scaler.scale_all` (regenerate derived packs from 2K).

## Conventions you will get wrong without this file

- **Template-action return values are `2`/`1`, NOT booleans.** `routine`, `routine_only_find`, high-level template/coordinate actions, and worker flow checks use `2` = success / found / clicked and `1` = not found / keep trying. `find_win(title)` is the exception: it returns `(left, top, width, height)` on success and `None` on failure; callers must test `is None` before unpacking.
- **Run/stop state is per-worker, not global.** Each `BaseWorker` holds `_active` (bool) + `_stop_event` (`threading.Event`). `_running()` = `_active and not _stop_event.is_set()` (this replaces the old `guaji_1`/`guaji_2` == 1 globals + `stop_event_1`/`stop_event_2`). `start()` (overridden) resets both so one instance can be restarted; `stop()` (called by the GUI stop button) sets both - safe on a not-started/finished thread. The worker sets `_active = False` itself on internal stop (timeout / LP exhaustion). The old `2`/`1`/`3` run-flag globals are gone; the `2`/`1` *return-value* convention (separate, above) is unchanged.
- **Template numbering - filename uses the *path* arg, not the *name* arg.** `click_item_with_result(self, picture, name)` / `find_item_with_result(self, picture, name)` try `<picture>_1.png`, `<picture>_2.png`, … incrementing until a file does **not** exist, trying each in turn. To add a fallback variant, drop in `<picture>_<next>.png` (where `<picture>` is the exact string passed as the first arg, e.g. `./aim/quests/link_raid/backup_requests/refresh_1.png`); the count is auto-discovered. The `name` arg is **only a log label** and never affects the filename. Match threshold `0.8` is hardcoded in `src/click/click_behavior.py` (`routine` / `routine_only_find`).
- **GUI -> worker parameters.** Parameters are **registry-driven, not globals**. Each worker class is decorated with `@register(name, label, params=[ParamSpec(...)], start_hint=...)` in `src/workers/`. The GUI iterates `get_registry()`, builds a widget per `ParamSpec` (by `kind`: `choice`->radio group, `lp_recover`->min/-1/input/+1/max, `int`->plain input), and on start reads each param's getter and `setattr`s it onto the worker (`worker.level_choice` / `worker.lp_recover_times`) before `worker.start()`. **There are no module-level parameter globals anymore** (the old `link_raid_lv_choice` / `link_raid_lp_recover_times` / `crystalis_lp_recover_times` are gone). Run/stop is via `worker.stop()` / the worker's own `_active` flag.
- **LP-recover values are "drinks + 1".** The GUI shows the *drink count* (link raid 0–10, crystalis 0–8), but the value passed to the worker is `shown + 1` (the `lp_recover` getter adds 1; e.g. default link raid `0` shown -> `1` stored = 0 drinks; default crystalis `8` shown -> `9` stored = 8 drinks). The worker decrements it once per depletion and stops when it hits `0`. So `1` means "stop on first depletion", not "one drink". The `ParamSpec.default` for `lp_recover` is the **shown** value (drinks), not the stored value.

## Template packs & the `aim` junction

- `aim/` at the repo/exe root is a **Windows directory junction** (reparse point), NOT a real folder. It points to one of the real packs under `language/`. All runtime paths use `./aim/...` and are transparently resolved through the junction.
- Real packs live at `language/<LANG>/<LANG>_<WxH>/` (e.g. `language/EN/EN_2560x1440`). The junction target is always one of these; pack folders are never moved or renamed.
- Both source packs are populated: `language/EN/EN_2560x1440` and `language/JP/JP_2560x1440` each contain the full EN/JP template tree (`crystalis/` + `quests/link_raid/...`). The derived packs (`*_1280x720`, `*_1920x1080`, `*_3840x2160`) start **empty** (`.gitkeep` only) and are filled by 刷新列表 (downscaled from 2K, except 4K which is upscaled).
- Switching language/resolution is done **in the GUI** (`LanguageSwitcherWidget` -> `language_switcher.switch(lang,res)`): selecting in the dropdown **auto-switches** - no apply button, only 刷新列表. Auto-switch fires on `QComboBox.activated` (user picks from the popup; programmatic `setCurrentIndex` during refresh is blocked with `blockSignals`, so repopulation never triggers a switch). It deletes the `aim` junction (`os.rmdir`, link-only) and recreates it with `cmd /c mklink /J` - no admin privileges. A pack is only switchable if it contains ≥1 `.png` (`pack_usable`); empty packs show as `（空）` and the switch is rejected.
- The language dropdown shows Chinese names via `language_switcher.lang_label(code)` (`EN`->英语, `JP`->日语), backed by `LANG_LABELS` in `src/packs/language_switcher.py`; item *data* holds the real code. **Adding a new language requires adding its Chinese label to `language_switcher.LANG_LABELS`** (NOT to the GUI), otherwise the bare code shows.
- `language/active.json` records `{lang, res}`. On startup `ensure_active()` restores `aim`: valid junction -> keep; missing/broken -> try `active.json`, else the first usable pack by scan order.
- **Do NOT** manually rename/move `aim/` or merge pack contents - it is a managed junction. To add a template variant, edit the `*_2560x1440` source pack; other-res packs are *derived* by `src/packs/image_scaler.py` on 刷新列表. To force a stale derived file to regenerate, delete it. The 1.5x `*_3840x2160` pack is the only *upscaled* (non-integer) pack and is slightly less stable; the 0.5x `*_1280x720` and 0.75x `*_1920x1080` packs are *downscaled* and clean.

## Empty pack directories are placeholders - do NOT delete the `.gitkeep` files

- `language_switcher.switch()` and `image_scaler.scale_pack()` **do NOT create missing pack directories**. `switch()` requires `os.path.isdir(target)` (returns False if the pack dir is absent); `scale_pack()` iterates `os.listdir(lang_dir)` and only processes **existing** subdirs (`_run_magick`'s `os.makedirs` only fires when writing a file inside a dir already in the loop).
- Therefore every empty resolution pack (`*_1280x720`, `*_1920x1080`, `*_3840x2160`) is kept alive by a committed **`.gitkeep`** so git preserves the directory. Without them, a fresh clone has no derived-pack dirs -> the GUI dropdown omits those resolutions and `image_scaler` can't populate them.
- `.gitkeep` is not a `.png`, so `pack_usable` still treats the dir as empty (`（空）`) until real templates (or scaled output) land - correct behavior. Don't "clean up" these files.

## Runtime prerequisites

- `resource/main.ico` lives at the **repo/exe root** (not inside the packs); `main.py` loads it via `./resource/main.ico`. Moving it out of the packs means `src/packs/image_scaler.py` no longer touches it.
- Game window title is hardcoded as `'MadokaExedra'` in `main.py`'s pre-start validation and in `src/click/click_action.py` / `src/click/click_behavior.py`. Changing it requires updating all three locations or centralizing a shared constant. All window-relative coordinate math depends on this lookup.
- Game must run **windowed at 16:9, at the same resolution as the selected template pack** (720p/1080p/2K/4K), with the matching template language; the game window must be foreground (the bot can be behind other windows, but the game cannot). Before starting a task, the GUI restores/focuses the game and refuses to start if the window cannot be found or has invalid geometry. It also runs `click_action.detect_window_resolution()` to compare the window's client area against the active pack's nominal resolution with a tolerant margin (title-bar/border/DPI drift allowed); a resolution mismatch only logs a warning and does **not** block the start. The JP server can be switched to EN in-game to reuse the EN templates instead of re-capturing.

## Bot modes & start conditions

Two independent loops, each with a required starting screen (the bot cannot navigate there itself):
- **Link raid** (`LinkRaidWorker` in `src/workers/link_raid.py`): start at the game's main/quest (灯台) screen. The first navigation step (`into_link_raid`) uses **hardcoded window-relative coordinates** (`click_position_scaled(2000,1000)` then `click_position_scaled(2400,1200)` - 2K baselines), NOT image recognition. `click_action` reads the active pack's resolution via `language_switcher.current_selection()` + `image_scaler.scale_factor()` and scales these 2K baselines by the factor (0.5/0.75/1.0/1.5), so the first click works at all resolutions; the scrolling drags `move_a_to_b_scaled(...)` scale the same way. Level range lv6–lv12 (default lv6).
- **Crystalis** (`CrystalisWorker` in `src/workers/crystalis.py`): start at the team-select screen (the "click play to enter battle" screen). Loops `click_play -> wait_win (find result/retry) -> click_retry_or_recover_lp`.

## Git / .gitignore

- `.gitignore` ignores: `aim/` (runtime junction), `language/active.json` (per-run config), `__pycache__/`, `*.pyc`. **Derived resolution packs are NOT gitignored**; their only committed content is `.gitkeep`, so pngs generated by `image_scaler` appear as untracked - commit or discard manually.
- `tools/ImageMagick/` is **committed to git** (runtime files only: `magick.exe`, `CORE_RL_*.dll`, VC/MFC runtime DLLs `msvcp140*`/`vcomp140`/`vcruntime140*`/`mfc140u`, `modules/coders` & `modules/filters`, config `*.xml`, `sRGB.icc`, `License.txt`). The scaler sets `MAGICK_HOME`/`MAGICK_CONFIGURE_PATH`/`MAGICK_CODER_MODULE_PATH`/`MAGICK_FILTER_MODULE_PATH`/`PATH` per-subprocess (without them it errors `CoderModulesPath` / `no decode delegate`); if `magick.exe` is absent it falls back to a `magick` on `PATH` with env untouched.
- Direct commits to `main`; PRs merged from forks. **Commit messages are plain-language Chinese** - a short descriptive subject and a plain-language body; favor everyday wording over `feat:`/`fix:` prefixes and jargon.

## README & multi-language docs

- `README.md` is the **Chinese source of truth** (the default). `README_EN.md` and `README_JP.md` are **derived translations**; when content changes, edit the Chinese one first, then mirror the change into the EN/JP copies. Never let the translations drift ahead of the Chinese original.
- The opening `<div align="center">` block (title + subtitle + badges + language switcher) is **pure HTML, not Markdown**: content inside an HTML block is not parsed as Markdown on strict CommonMark renderers, so a Markdown `# Title` renders as literal text instead of an `<h1>` (the title looks unstyled). Therefore use `<h1>` for the title, `<p>`/`<strong>` for the subtitle, `<img>` for badges, and `<a>` for the language-switcher links. **Never revert the title back to `#` or the links back to `[...](...)` Markdown syntax** - that reintroduces the bug.
- All three files carry the same language switcher, centered at the top of the `<div align="center">` block (after the title and badges):
  ```html
  <a href="./README.md">简体中文</a> · <a href="./README_EN.md">English</a> · <a href="./README_JP.md">日本語</a>
  ```
  Keep it identical across the three files (same position, same text) and point links at `./README*.md` (repo-relative).
- **Do not translate:**
  - Usernames / contributor names (e.g. `TIAN000000`, `智谱AI`) - leave them verbatim in every language. (One-off name tweaks are exceptions, not a translation rule.)
  - Game mode proper nouns - `Link Raid`, `Crystalis` (the Chinese README calls the latter 晶花; the EN/JP copies use `Crystalis`, matching the code/template folder `crystalis`, to avoid ambiguity).
  - Library names & code identifiers - `OpenCV`, `TM_SQDIFF_NORMED`, `PyAutoGUI`, `PySide6`, `pywinctl`, `ImageMagick`, `MadokaExedra`, `GLM-5.2`, `backup_requests`, all `*.py` / `*.md` filenames, junction identifiers, etc.
  - **GUI button text** - the GUI is Chinese-first, although exact visible labels may contain technical identifiers or proper nouns such as `Link Raid`, `play`, and `pack`. In EN/JP copies, keep the original visible label verbatim (e.g. `link raid挂机启动`, `自动刷晶花，需要在play界面启动`, `刷新列表`, `检查更新`), optionally followed by a short gloss in the target language, so readers can find the actual control on screen.
- **Do translate** the surrounding prose. When leaving a technical term untranslated would make a sentence read awkwardly, choose the well-established target-language term deliberately (e.g. junction -> ジャンクション, threshold -> しきい値, template -> テンプレート, match -> マッチング, screenshot -> スクリーンショット) rather than forcing an awkward literal copy.
- `README_EN.md` and `README_JP.md` each end with an **AI-translation disclaimer** (in their own language) stating the page is AI-translated, may contain ambiguities/inaccuracies, and that the Chinese `README.md` is authoritative. The Chinese `README.md` has **no** such disclaimer (it is the source).
