# 自動更新とリリース

言語：[简体中文](./首页) · [English](./Home_EN) · [日本語](./ホーム)

## バージョン番号

- バージョン番号は明示的な3段落 SemVer を使用します（例：`2.2.0`、`2.2.0-beta`）
- `VERSION`（`src/update_check.py`）に `v` プレフィックスはありません。タグは正確に `v{VERSION}` です
- stable にはプレリリース接尾辞がなく、`prerelease=false` です
- beta は確認された接尾辞を使用し、`prerelease=true` です
- VERSION、タグ、Release の `tag_name`、アセットタグフラグメントはすべて一致する必要があります

## 更新確認

`src/update_check.py` は GitHub Releases API で最新リリースを照会し、ローカルの `VERSION` とセマンティックバージョン比較を行います。

- ダウングレード防止：リモートバージョンがローカルバージョンより厳密に大きい場合のみ「更新あり」とします
- リモートがローカル以下の場合は常に「最新です」と報告し、ダウングレードを提示しません
- 照会失敗（ネットワークなし、API レート制限、解析エラー）は例外を送出せず、`has_update=False` とメッセージを返します

### 更新チャネル

- **stable チャネル**（デフォルト）：正式リリースのみ参照（`releases/latest` はプレリリースを除外）。beta は提示しません
- **beta チャネル**：すべてのリリース（プレリリース含む）を参照し、semver で最大のものを選択します。プレリリースバージョンは semver ルールで比較します（正式版 > 同バージョンのプレリリース）

ソースモードでは自動確認は Release URL のみを記録し、手動確認のみがそれを開きます。

## 安全なインストール

自動インストールには以下が必要です：

- GitHub SHA-256
- キャンセル可能なサイズ/ハッシュチェック付きダウンロード
- 安全な境界付き展開
- AMD64 PE 検証
- バックアップ
- ハッシュ検証
- ロールバック/リカバリマーカー
- 起動ヘルスハンドシェイク

読み取りは1つのユニークな `MagiaExedra_auto_<tag>.zip` または `MagiaExedra_auto_<tag>_win64.zip` と互換性を維持します。新リリースは `_win64` 接尾辞を使用する必要があります。

## アセット命名

アセット名は厳格です：

- stable：`MagiaExedra_auto_v<version>_win64.zip`
- beta：`MagiaExedra_auto_v<version-with-prerelease>_win64.zip`

## リリース手順

1. まずリモートのブランチ/タグをフェッチし、同名のタグ、Release、アセットが既存か確認します。公開済みタグは絶対に移動させません
2. 祖先リリース基準を選択して記録します。stable は通常前の stable 祖先を使用し、beta は前の beta 祖先を使用し、事前 beta がない場合は最近の stable 祖先を使用します
3. 基準からリリースコミットまでのすべてのコミットと完全な diff をレビューします
4. `src/update_check.py::VERSION` とバージョン関連ドキュメントを更新し、該当する非ゲームチェックをすべて実行します
5. `pyinstaller -D -i resource/main.ico main.py` で新規の出力/ステージングディレクトリにビルドします。古い非追跡の `main.spec` や `dist/` は再利用しません
6. 許可リストで組み立てます：新規の `main.exe` と `_internal/`、 plus `resource/`、`language/`、`tools/` 配下のバージョン管理された実行時ファイル。`main.exe` は ZIP のルートに配置します。`aim/`、`active.json`、`.source_hashes.json`、ローカル生成の派生 PNG、キャッシュ、Git/ビルドメタデータ、無関係な非追跡ファイルを除外します
7. ZIP CRC/エントリ、追跡された pack プレースホルダ、実行時/生成ファイルなし、AMD64 PE を検証します。最終 ZIP に対して現在の `extract_update()` を実行し、ルート/manifest を確認します。`find_asset()` が期待されるメタデータを一意に識別することを確認します
8. 最終ステージングツリーの使い捨てコピーから `main.exe` をスモークスタートし、即座に終了せず Qt アプリの作成/worker の読み込みができることを確認します。最終 ZIP の前にスモークで作成された `aim/` と `active.json` を削除します
9. まず対象ブランチをプッシュします。**ドラフト Release** を作成し、1つの最終 ZIP をアップロードし、`state=uploaded` と GitHub ダイジェストを待ってから、アセット名、バイトサイズ、SHA-256 をローカル値と比較します。公開前には再ダウンロードして再確認することを推奨します
10. すべてのチェックが通った場合のみ公開します。タグ/Release は正確にプッシュされたコミットを指す必要があります。beta はプレリリースでなければならず、latest にはなりません。失敗時はドラフトのままにするか、不良アセット/Release を削除します。公開済みタグの下でコンテンツを黙って置き換えることは絶対にしません

## Release 説明の形式

各 Release 説明は以下のセクションを順番に使用します：

```markdown
## 主要更新

- ユーザー向けの変更、重要なセキュリティ修正、動作変更

## 构建信息

- バージョン、Git タグ、対象ブランチ
- Windows アーキテクチャ、Python バージョン、PyInstaller バージョン
- アセットファイル名、バイトサイズ、SHA-256

## 测试说明

- 実際に実行したチェックと結果
- （beta のみ）未完了の全フロー検証の範囲と残余制限
```

Release 説明は確認された基準からリリースまでの範囲に固有の内容にする必要があります。beta Release のみがテスト説明に未完了の全フロー検証に関する記述を含めます。stable Release にはこのような記述を含めず、テスト説明には実際に実行したチェックのみを記載します。

## リリース後の報告

公開後の報告：ブランチ、完全なコミットハッシュ、タグ、stable/プレリリース状態、Release URL、アセット名/サイズ/SHA-256、実行したチェック、残余制限、非追跡のローカルビルド産物。
