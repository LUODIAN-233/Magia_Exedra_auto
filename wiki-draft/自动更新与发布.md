# 自动更新与发布

语言：[简体中文](./首页) · [English](./Home_EN) · [日本語](./ホーム)

## 版本号约定

- 版本号使用显式三段式 SemVer，如 `2.2.0`、`2.2.0-beta`
- `VERSION`（`src/update_check.py`）不带 `v`；tag 恰好是 `v{VERSION}`
- stable 无预发布后缀，`prerelease=false`
- beta 使用确认的后缀，`prerelease=true`
- VERSION、tag、Release 的 `tag_name`、资产 tag 片段必须一致

## 更新检查

`src/update_check.py` 通过 GitHub Releases API 查询最新发布版本，与本地 `VERSION` 做语义化版本比较。

- 防降级：只有远端版本严格大于本地版本时才视为「有更新」
- 远端版本小于或等于本地时一律报告「已是最新」，绝不提示降级
- 查询失败（无网络、API 限流、解析失败）不抛异常，只返回 `has_update=False` 并附说明

### 更新通道

- **stable 通道**（默认）：只看正式 release（`releases/latest` 已排除预发布），不会提示 beta
- **beta 通道**：看所有 release（含预发布），选 semver 最大的，可能含 beta。预发布版本按 semver 规则比较（正式版 > 同版本预发布）

源码模式下自动检查只记录 Release URL，只有手动检查才打开它。

## 安全安装

自动安装要求：

- GitHub SHA-256
- 可取消的大小/哈希校验下载
- 安全有界解压
- AMD64 PE 校验
- 备份
- 哈希验证
- 回滚/恢复标记
- 启动健康握手

读取仍兼容唯一的 `MagiaExedra_auto_<tag>.zip` 或 `MagiaExedra_auto_<tag>_win64.zip`；新发布必须使用 `_win64` 后缀。

## 资产命名

资产名称严格规定：

- stable：`MagiaExedra_auto_v<version>_win64.zip`
- beta：`MagiaExedra_auto_v<version-with-prerelease>_win64.zip`

## 发布流程

1. 先 fetch 远端分支/tag，检查是否已存在同名 tag、Release 或资产。绝不移动已发布的 tag
2. 选择并记录祖先发布基线。stable 通常用上一个 stable 祖先；beta 用上一个 beta 祖先，若无先前 beta 则用最近的 stable 祖先
3. 审查从基线到发布提交的每个 commit 和完整 diff
4. 更新 `src/update_check.py::VERSION` 和版本相关文档，然后执行所有适用的非游戏检查
5. 用 `pyinstaller -D --windowed -i resource/main.ico -n Magia_Exedra_auto main.py` 构建到全新输出/暂存目录，不要复用旧的非跟踪 `main.spec` 或 `dist/`
6. 按允许清单组装：新的 `Magia_Exedra_auto.exe` 和 `_internal/`，加上 `resource/`、`language/`、`tools/` 下的版本控制运行时文件。`Magia_Exedra_auto.exe` 必须在 ZIP 根。排除 `aim/`、`active.json`、`.source_hashes.json`、本地生成的派生 PNG、缓存、Git/构建元数据以及无关的非跟踪文件
7. 验证 ZIP CRC/条目、跟踪的 pack 占位、无运行时/生成文件、AMD64 PE。对最终 ZIP 运行当前 `extract_update()` 并确认其根/manifest。确认 `find_asset()` 唯一标识预期元数据
8. 从最终暂存树的一次性副本冒烟启动 `Magia_Exedra_auto.exe`，确认它不会立即退出且能创建 Qt app/加载 workers。最终 ZIP 前删除冒烟创建的 `aim/` 和 `active.json`
9. 先推送目标分支。创建**草稿 Release**，上传一个最终 ZIP，等待 `state=uploaded` 和 GitHub 摘要，然后比较资产名、字节数和 SHA-256 与本地值。发布前优先重新下载并复查
10. 所有检查通过后才发布。tag/Release 必须指向精确的推送提交。beta 必须是预发布且永远不是 latest。失败时保持草稿或删除坏的资产/Release；绝不在已发布 tag 下静默替换内容

## Release 说明格式

每个 Release 说明按顺序使用以下章节：

```markdown
## 主要更新

- 面向用户的改动、重要安全修复和行为变化

## 构建信息

- 版本、Git tag 和目标分支
- Windows 架构、Python 版本和 PyInstaller 版本
- 资产文件名、字节数和 SHA-256

## 测试说明

- 实际执行的检查及结果
- （仅 beta）未完成全流程测试的范围与残余限制
```

Release 说明必须针对确认的基线到发布范围具体撰写。只有 beta Release 才在测试说明中包含未完成全流程测试相关话术；stable Release 不写此类提示，测试说明只列实际执行的检查。

## 发布后报告

发布后报告：分支、完整 commit 哈希、tag、stable/预发布状态、Release URL、资产名/大小/SHA-256、执行的检查、残余限制以及非跟踪的本地构建产物。
